# GTV 评估维度详解

本文档详细描述每个评估维度的具体评分要素，供人工评分或 API 不可用时参考。

---

## 一、教育背景（MC，权重 25%）

### 院校排名梯度

| 梯队 | 典型院校 | 加分 |
|------|----------|------|
| 顶尖 | QS/THE 前 20、清北复交、MIT/Stanford/Oxbridge 等 | 满分档 |
| 一流 | QS 前 50、985/211 头部、北美 Top 30 | 高分段 |
| 知名 | QS 前 100、211、北美 Top 100 | 中高分段 |
| 普通 | 其他本科院校 | 基础分 |
| 未列名 | 无排名或信息缺失 | 酌情扣分 |

### 学历层次加分

| 学历 | 基础加分 | 说明 |
|------|----------|------|
| 博士 | 最高 | 研究型路径尤其重要 |
| 硕士 | 高 | 技术/工程路径常见 |
| 本科 | 中 | 需配合强工作经历 |
| 其他 | 低 | 需突出实践成就 |

### 专业匹配度

- **高度相关**：CS、EE、软件工程、数据科学、AI/ML 等与申请领域直接对应
- **相关**：数学、物理、自动化等可支撑技术申请
- **弱相关**：需额外证明技能迁移和行业贡献

## 二、工作经验（MC，权重 30%）

### 公司梯度

| 梯队 | 典型公司 | 说明 |
|------|----------|------|
| 顶尖 | FAANG、微软、英伟达等全球科技龙头 | 最高认可度 |
| 一线 | 知名独角兽、头部互联网、顶尖金融机构科技部门 | 高认可度 |
| 二线 | 上市公司、成长型科技公司 | 中高认可度 |
| 三线 | 中小科技公司、创业公司 | 需结合成就评估 |
| 其他 | 传统行业、外包、初创早期 | 需强证据支撑 |

### 职位级别映射

| 级别 | 典型职位 | 年限参考 |
|------|----------|----------|
| 高层 | CTO、VP Engineering、Principal/Staff | 8+ 年 |
| 资深 | Senior、Lead、Architect | 5–8 年 |
| 中级 | Mid-level、Engineer II/III | 3–5 年 |
| 初级 | Junior、Engineer I | 1–3 年 |

### 年限与权重

- 5+ 年且公司/职位层级高：可给满分档
- 3–5 年且公司知名：高分段
- 2–3 年：中分段，需其他维度补强
- 1–2 年：低分段，除非成就突出

## 三、技术专长（MC，权重 20%）

### 认证含金量排名（示例）

| 层级 | 认证类型 | 示例 |
|------|----------|------|
| 高 | 厂商核心认证、顶级竞赛 | AWS Solutions Architect Professional、ACM-ICPC |
| 中 | 专业认证、行业标准 | Kubernetes CKA、PMP、各类云认证 |
| 低 | 入门认证、培训证书 | 各类入门级证书 |

### 技术栈深度 vs 广度

- **深度**：在 1–2 个核心领域有深入实践（架构、性能、安全等）
- **广度**：跨多个技术栈，能支撑全栈或跨域项目
- **理想**：核心领域深度 + 适度广度

### 证明方式

- 开源项目（stars、fork、贡献者）
- 专利、软著
- 技术博客、演讲、培训
- 认证证书

## 四、行业影响力（MC，权重 25%）

### 不同类型贡献的权重

| 类型 | 权重 | 说明 |
|------|------|------|
| 行业奖项 | 高 | 官方/权威机构颁发 |
| 媒体报道 | 高 | 主流媒体、专业媒体 |
| 标准制定 | 高 | 参与国标、行标、开源标准 |
| 会议演讲 | 中高 | 级别越高权重越大 |
| 项目影响 | 中 | 用户量、收入、社会效益 |
| 内部认可 | 低 | 公司内部奖项，需结合外部证据 |

### 量化指标

- 用户/客户规模
- 收入/增长数据
- 市场份额
- 政府/机构合作
- 专利/论文引用

## 五、国际认可度（OC，权重 30%）

### 会议级别

| 级别 | 说明 | 示例 |
|------|------|------|
| A 类 | 顶会、旗舰会议 | NeurIPS、ICML、OSDI、SIGCOMM |
| B 类 | 知名会议 | 各领域主流会议 |
| C 类 | 一般会议 | 区域会议、workshop |

### 奖项含金量

- **国际顶级**：图灵奖、菲尔兹奖、各领域顶级奖项
- **国际知名**：领域内国际奖项、竞赛金奖
- **国家级**：政府科技奖、行业大奖
- **其他**：公司内部、区域奖项

### 跨国项目

- 跨国团队协作
- 海外客户/合作
- 国际标准参与
- 海外分支/派驻经历

## 六、创新贡献（OC，权重 25%）

### 专利类型差异

| 类型 | 含金量 | 说明 |
|------|--------|------|
| 发明专利 | 高 | 技术方案完整、有审查 |
| 实用新型 | 中 | 结构/方法改进 |
| 软著 | 低 | 软件著作权，证明原创即可 |

### 开源项目评估标准

| 指标 | 说明 |
|------|------|
| Stars/Forks | 关注度与使用度 |
| 贡献者 | 社区参与度 |
| 维护时长 | 持续投入 |
| 技术深度 | 是否解决实际问题 |
| 领域相关性 | 与申请领域是否匹配 |

### 技术突破

- 性能/效率显著提升
- 新方法/新架构
- 可复现、可验证的成果

## 七、领导力（OC，权重 25%）

### 团队规模映射

| 规模 | 说明 |
|------|------|
| 20+ 人 | 大型团队，高层管理 |
| 10–20 人 | 中型团队，技术+管理 |
| 5–10 人 | 小团队负责人 |
| 2–5 人 | 项目组/小组领导 |
| 1 人 | 个人贡献者，领导力证据弱 |

### 技术决策层级

- **战略级**：技术路线、架构选型、重大投入
- **战术级**：项目技术方案、技术栈选型
- **执行级**：具体实现、代码审查

### Mentor 经历

- 带新人、实习生
- 内部技术分享、培训
- 外部技术社区指导

## 八、社会影响（OC，权重 20%）

### 开源贡献量化

| 指标 | 说明 |
|------|------|
| 项目数量 | 主导或深度参与的项目数 |
| 代码行数/PR 数 | 贡献量 |
| 社区角色 | Maintainer、Core contributor |
| 影响范围 | 用户数、被引用/依赖情况 |

### 社区影响力指标

- 技术博客阅读量
- 演讲/分享次数
- 社交媒体技术影响力
- 开源社区职位（如 PMC、Committer）

### 社会价值

- 教育/培训项目
- 公益技术项目
- 政府/公共部门合作
- 可量化的社会效益
