# GTV 证据描述写作指南

## 标准描述结构：What → Why → How

每份证据的描述应采用三段式结构，帮助评审官快速理解证据的价值和与标准的对应关系。

| 部分 | 内容 | 字数建议 |
|------|------|----------|
| **What**（是什么） | 证据的客观描述：类型、名称、时间、主体 | 50-100 字 |
| **Why**（为什么重要） | 该证据对 GTV 申请的意义、行业背景 | 50-100 字 |
| **How**（如何证明标准） | 明确对应 MC/OC 的哪个维度，关键数据支撑 | 100-300 字 |

### 示例（项目证据）

```
【What】本项目为 XX 公司于 2022-2024 年开发的智能风控平台，申请人担任技术负责人。

【Why】该平台服务 XX 家金融机构，是行业内领先的 AI 风控解决方案，体现了申请人在数字技术领域的领导角色。

【How】对应 MC2（产品/平台领导角色）和 OC1（创新贡献）。平台日处理交易量 XX 万笔，准确率 XX%，较传统方案提升 XX%。申请人主导架构设计和技术选型，带领 XX 人团队完成核心模块开发。
```

## 不同证据类型的描述模板

### 项目证据

```
【What】[项目名称]，[时间范围]，[申请人角色]，[项目类型/规模]。

【Why】[行业地位/市场意义/技术难度]。

【How】对应 [MC/OC 编号]。[关键量化数据：用户量/交易量/性能指标/团队规模]。[申请人具体贡献]。
```

### 专利证据

```
【What】[专利名称]，[专利号]，[申请/授权日期]，[申请人作为第几发明人]。

【Why】[技术领域/创新点/与申请人专业的关联]。

【How】对应 OC2（学术/创新贡献）。[专利解决的技术问题]、[应用场景]、[已产生的商业/学术影响]。
```

### 论文证据

```
【What】[论文标题]，发表于 [期刊/会议名称]，[时间]，[申请人角色：第一作者/通讯作者等]。

【Why】[期刊/会议级别]，[引用次数/下载量等]。

【How】对应 OC2。[研究贡献]、[方法创新]、[与申请领域的关联]。
```

### 奖项证据

```
【What】[奖项名称]，[颁发机构]，[时间]，[申请人获得的具体荣誉]。

【Why】[奖项的行业地位/评选标准/竞争激烈程度]。

【How】对应 MC1（专业成就与认可）。[评选维度]、[申请人脱颖而出的原因]、[后续影响]。
```

### 媒体报道证据

```
【What】[媒体名称]，[报道标题]，[发布时间]，[报道类型：专访/专题/新闻]。

【Why】[媒体级别/影响力]、[报道角度]。

【How】对应 MC1 或 MC3。 [报道中引用的关键事实和数据]、[对申请人/公司的评价]。
```

### 开源贡献证据

```
【What】[项目名称/仓库]，[时间范围]，[申请人角色：核心贡献者/维护者]。

【Why】[项目规模：stars/forks/用户数]、[在生态中的位置]。

【How】对应 OC1 或 OC3。[具体贡献：代码量/PR 数/模块]、[社区认可]、[技术影响力]。
```

### 行业演讲证据

```
【What】[会议/活动名称]，[时间]，[地点]，[申请人演讲主题]，[听众规模/级别]。

【Why】[会议级别：国际/国内/行业]、[同场演讲者]。

【How】对应 MC1 或 OC3。[演讲内容的技术深度]、[反馈/影响]、[体现的行业地位]。
```

## 描述长度控制

| 证据类型 | 建议字数 | 说明 |
|----------|----------|------|
| 核心证据（MC 关键支撑） | 400-500 字 | 充分展开，数据详实 |
| 重要证据（OC 或 MC 辅助） | 300-400 字 | 平衡详略 |
| 一般证据 | 200-300 字 | 简明扼要 |

**单份上限**：不超过 500 字，避免喧宾夺主。

## 与 MC/OC 标准的对应关系

### Mandatory Criteria（MC）

| 编号 | 标准 | 证据方向 | 描述中需体现 |
|------|------|----------|--------------|
| MC1 | 专业成就与认可 | 奖项、媒体报道、行业认可 | 认可来源、评选标准、竞争程度 |
| MC2 | 产品/平台领导角色 | 组织架构、产品、领导证明 | 角色、职责、团队规模、产品影响 |
| MC3 | 商业成功 | 市场报道、收入、用户规模 | 量化数据、增长指标 |
| MC4 | 行业影响力 | 数据统计、政府认可、市场份额 | 规模、覆盖范围、权威认可 |

### Optional Criteria（OC）

| 编号 | 标准 | 证据方向 | 描述中需体现 |
|------|------|----------|--------------|
| OC1 | 创新贡献 | 创新产品、技术突破 | 创新点、技术难度、应用效果 |
| OC2 | 学术贡献 | 论文、专利、学术认可 | 发表级别、引用、专利价值 |
| OC3 | 技术领导力 | 技术决策、团队、项目 | 决策内容、团队成果、项目影响 |
| OC4 | 行业外贡献 | 社会影响、跨界合作 | 社会价值、跨界范围 |

### 描述中的对应标注

每份证据描述**必须**明确写出对应的 MC/OC 编号，例如：

- 「对应 MC2（产品/平台领导角色）和 OC1（创新贡献）」
- 「满足 MC1 对专业认可的要求」
- 「体现 OC3.1（技术决策能力）」

## 写作禁忌

| 禁止项 | 说明 |
|--------|------|
| 无量化数据 | 尽量包含数字：用户量、收入、团队规模、百分比等 |
| 模糊对应 | 必须明确写出 MC/OC 编号，而非泛泛说「证明能力」 |
| 重复论证 | 同一数据点不在多份证据中重复展开 |
| 主观评价 | 避免「重要的」「显著的」，用事实和数据说话 |
| 超长描述 | 单份超过 500 字需精简 |

## 证据描述文件格式建议

多份证据可合并为一个文件，用分隔符区分：

```
========== 证据 1：XX 项目 ==========
[描述内容]

========== 证据 2：XX 专利 ==========
[描述内容]

========== 证据 3：XX 奖项 ==========
[描述内容]
```

或按证据编号分文件：`evidence_MC2_1.txt`、`evidence_OC1_1.txt` 等。
